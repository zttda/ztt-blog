# 博客日常维护规则

这份规则只保留个人博客真正需要的步骤。平时优先使用 `open-blog-panel.bat`，不需要记住所有命令。

## 每次开始

1. 确认上一次修改已经处理完；面板里的“Git 状态”不应出现看不懂的大批文件。
2. 换电脑或多人修改时，先用 `git pull --ff-only` 获取远端更新。
3. 不打开或提交 `.tmp-*`、浏览器资料、`node_modules`、`dist`、设计候选图和密钥文件。

## 写文章

- 新文章保持“先隐藏”勾选，也就是 `draft: true`。
- 标题说明主题，摘要用一句话说明读者会得到什么，标签优先复用已有标签。
- 图片必须写有意义的说明；正文图优先 WebP/AVIF，建议宽度不超过 1600px、单张不超过 1 MiB。
- 草稿也可能随 Git 上传到 GitHub。草稿不能保存密码、token、身份证号、私人聊天或未授权素材。
- 修改已发布文章时填写 `updatedDate`，避免读者误以为内容从未变更。

## 发布前检查

1. 在本地预览首页、文章页和手机宽度，确认文字不重叠、图片正常、链接可打开。
2. 点击“构建检查”，必须成功后再发布。
3. 查看变更文件：文章发布通常只应涉及文章、实际使用的图片和少量配置。
4. 如果突然出现几百个文件、浏览器目录或超过 20 MiB 的文件，停止发布并先排查。
5. 提交说明不要写笼统的 `update blog`，使用 `post:`、`site:` 或 `ops:` 开头说明目的。

## 发布后检查

1. 等待 Cloudflare Pages 显示部署成功。
2. 打开线上首页和本次文章，确认域名、图片、样式和移动端显示正常。
3. 失败时不要反复修改无关文件；先查看构建日志里的第一条实际错误。

## 回滚

- 文章内容有问题：先把文章改回 `draft: true` 并重新发布，这是最快的止损方式。
- 代码或样式有问题：找到引入问题的提交，使用 `git revert <提交编号>` 生成反向提交，再推送。
- 不使用 `git reset --hard` 或强制推送处理普通事故；它们可能丢失本地和远端记录。
- Cloudflare Pages 可以临时回退到上一份成功部署，但仓库仍应随后用 `git revert` 修正。

## 备份与清理

- GitHub 是主备份；每月或重要改版前，再复制一次整个仓库到其他磁盘。
- 每月检查一次未引用图片、重复图片和草稿。删除素材前先确认页面与文章没有引用。
- 设计过程图放在仓库外的归档目录；仓库只保留最终使用的页面资源和必要验收图。
- 已推送的 Git 历史重写会影响远端和其他副本，必须先做完整备份并单独确认。
